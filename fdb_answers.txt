Q1: FDB separates its architecture into TMS and SS because it allows them to scale separately. This makes sense because TMS and SS need to be scaled in different situations. If the workload is write-heavy, then we would want to scale TMS, and the opposite for SS.
Q2: The Resolver is responsible for handling conflicts. This occurs before writing to LogServers as commits can't be finalized if there are conflicts.
Q3: The read version is assigned at the beginning of a transaction, so if the transaction fails or is slow to resolve, the order of the transactions is still denoted correctly by the read version, as opposed to the write version which is assigned at the end of a transaction. When a transaction commits, it checks for differences between the read and write version and aborts if there are differences. This allows FDB to provide strict serializability without locking.
Q4: Snapshot reads don't check for conflicts, so in certain situations where it's fine to have stale data (high write frequency workloads), snapshot reads allow for reads that may be aborted by standard reads. When real-time correctness is more important, standard reads should be used.
Q5: For each transaction, data is written to LogServers, and the transaction commits when the data is successfully replicated. Then, the data is pulled to StorageServers asynchronously. The TMS is stateless, so when it fails, recovery simply involves rebooting the TMS and then getting the latest commit point from the LS, as there's no need to replay any logs. Any inflight transactions will simply get aborted.
